Everything else is green or a nit. Once coverage reaches ≥ 95 % on `custom_components/nightscout_v3/api/*` and the two above items are addressed (or consciously waived and documented), Phase 1 can be approved.

## 3. Severity-ranked issues

### Critical (must fix before Phase 1 is considered done)

- **C-1. Coverage below 90 % CI gate.** `pyproject.toml:addopts` sets `--cov-fail-under=90`. Current combined coverage is 87.92 %, so `pytest` fails under its own rules. The spec aims for ≥ 95 %. Missing lines are listed in §4.
- **C-2. JWT exchange uses GET, spec says POST.** `api/auth.py:77` calls `self._session.get(...)`. Spec `§3.3.1:133` and reference `docs/references/nightscout-v3-api.md:13,19` both specify `POST`. The plan at `docs/plans/2026-04-22-ha-nightscout-v3-plan.md:829` coded `session.get`, so the drift originated in the plan and was inherited. Either:
  - change to `session.post(...)` (and update the 3 aioresponses mocks in `tests/test_auth.py` from `m.get` to `m.post`), or
  - confirm empirically that the target server accepts GET, update the spec + reference to say "GET" (preferred: GET *and* POST, documenting which), and note the deviation in the plan.

### Important (should fix; candidates for Phase 2 carry-over only if consciously deferred)

- **I-1. `probe_capabilities` raises `RuntimeError`, not `ApiError`.** `api/capabilities.py:58`. The review criteria explicitly require the three-way mapping `AuthError`/`ApiError`/`NotReady`. A coordinator later catching `ApiError` will miss this and surface an uncaught `RuntimeError`. Suggest `raise ApiError("Server exposes no entries; cannot proceed")`. Tests would change `pytest.raises(RuntimeError, match="entries")` → `pytest.raises(ApiError, match="entries")`.
- **I-2. `asyncio.create_task` + `asyncio.gather` leaks tasks on partial failure.** `api/capabilities.py:43-55` pre-creates seven tasks before `await asyncio.gather`. If one of the earlier tasks raises while later ones are still pending and `gather` receives `return_exceptions=False` (the default), the pending tasks continue running unobserved. Cleaner pattern: `asyncio.gather(client.get_status(), client.get_devicestatus(...), ...)` directly — `gather` schedules and cancels them atomically. Not a functional bug given the short timeouts, but it's a resource-hygiene issue the Silver scale will ask about.
- **I-3. `body.get("result") or {}` silently swallows a falsy but legitimate result.** `api/auth.py:90`. If the server ever returns `"result": {}` or `"result": null`, the code proceeds to `token = None` and only *then* raises "Malformed JWT response" — which is fine, but `or {}` obscures the root cause in the eventual error string. A pedantic `result = body.get("result", {})` is equivalent and less confusing.
- **I-4. `if not (token and exp and iat)` rejects `exp == 0` or `iat == 0`.** `api/auth.py:94`. In practice `iat`/`exp` are always positive epoch seconds, so this is theoretical — but the intent is "present", not "truthy". Prefer `if token is None or exp is None or iat is None`.

### Nits (take or leave)

- **N-1. `api/client.py` imports `AsyncMock` in the test file but never uses it.** `tests/test_client.py:4`, likewise `tests/test_capabilities.py:4` imports `AsyncMock` at module scope even though the pattern works without it in `test_capabilities.py`. Dead imports; `ruff` should catch these.
- **N-2. `test_auth.py:5` imports `AsyncMock` but never uses it.** Same as above.
- **N-3. `get_valid_jwt` holds the lock while doing the HTTP call.** `api/auth.py:49`. Correct for thundering-herd but means other consumers stall behind the full network timeout. For a single-coordinator setup it's fine; mentioning here because Phase 3 may want to remember it.
- **N-4. `get_entries(limit=1)` default is non-obvious.** The only caller in Phase 1 is `probe_capabilities` with an explicit `limit=1`. When Phase 2/3 uses `get_entries` for the 14-day window it must remember to pass `limit=1000`; consider adding a docstring hint. The sibling `get_treatments`/`get_devicestatus` have the same pattern, so it's internally consistent.
- **N-5. `get_entries` sort params are built in a weird order.** `api/client.py:57-61` uses `params.insert(0, …)` to prepend date filters. The order doesn't matter for the wire (query-param order is not semantic), but the test at `tests/test_client.py:61` relies on a specific URL string (`date$gte=…&limit=1000&sort$desc=date`), which is brittle — a future refactor to `.append` would break the test even though the behavior is unchanged. Consider either sorting params deterministically inside `_raw_get`, or letting the test match the full URL regardless of param order via a regex matcher.
- **N-6. Magic strings for event types in `capabilities.py`.** `_SENSOR_CHANGE`, `_SITE_CHANGE`, etc. are defined as module-private constants, which is fine, but Phase 2's careportal code will likely want the same strings. A shared `constants.py` (still pure-Python) would prevent drift. Not actionable in Phase 1.
- **N-7. `pyproject.toml` sets `--cov-fail-under=90`, spec says 95 %.** Align the CI gate with the spec.
- **N-8. `ApiError("No profile returned")` semantic mismatch.** `api/client.py:86`. Returning an empty list is a valid server response (no profile configured), not really an *error*. The coordinator layer will want to distinguish "server didn't have one" from "server is broken". Consider `return None` + `Optional[dict]` return type, or a dedicated `NoProfileError(ApiError)`.

## 5. Security / Silver-gate notes

- `aiohttp.ClientTimeout(total=30)` is applied uniformly — Silver Rule 82 (timeouts) satisfied.
- `asyncio.Lock` is held during refresh — Silver Rule 93 (no thundering-herd on auth) satisfied.
- No credentials are logged: `_LOGGER.debug` in `auth.py:71` logs only attempt number and sleep duration. Good.
- The `quote(v, safe='$')` in `client.py:105` correctly URL-encodes values while keeping Nightscout's `$` operators unescaped in keys *and* values. (Keys themselves come from hard-coded literals so injection is not a concern.)

- No `eval`, no `exec`, no `pickle`, no `shell=True` anywhere. Clean.
